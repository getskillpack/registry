# skpkg (Skill Manager) и классические менеджеры пакетов: сравнение для контента

Внутренняя справка для Growth/CMO. **Не** позиционировать как «skpkg лучше npm» — у экосистем разные предметные области и зрелость.

**Определения в этом документе**

- **skpkg** — экосистема [getskillpack](https://github.com/getskillpack): HTTP-реестр skill-пакетов (репозиторий `registry`) и клиент **`skillget`** (репозиторий `cli`, ядро на Go). Пакет = версионированный `.tar.gz` с манифестом и файлами скилла (например `SKILL.md`).
- **npm** — реестр и инструментарий для пакетов **JavaScript/Node.js** (публичный индекс npmjs.com и совместимые зеркала).
- **Packagist + Composer** — стандартный способ распространения библиотек **PHP**; метаданные на packagist.org, исходники чаще тянутся из Git/VCS по ссылкам, а не только как заранее залитый архив.
- **PyPI** — канонический индекс пакетов **Python** (форматы sdist/wheel, установка через `pip`, `uv`, Poetry и т.д.).

## Сводная таблица

| Измерение | skpkg | npm | Packagist / Composer | PyPI |
|-----------|-------|-----|----------------------|------|
| **Тип артефакта** | Tarball + JSON-манифест в API | Обычно tarball из реестра; метаданные `package.json` | Описание пакета + VCS URL / dist URL; установка из Git или zip | sdist / wheel в индексе |
| **Типичный сценарий** | Установка **набора инструкций/шаблонов для агентов** (skills) в среде разработки | Зависимости приложений и библиотек JS | Зависимости PHP-проектов | Зависимости Python-проектов |
| **Публичный индекс** | Зависит от развёрнутого реестра (публичный или корпоративный); контракт API — [registry-api.md](./registry-api.md) | Централизованный npmjs.com | packagist.org | pypi.org |
| **Модель доверия (кратко)** | Публикация с **Bearer-токеном** на стороне сервера; клиент получает **checksum** версии; есть **yank** версии. Нет встроенной модели «мейнтейнер npm» — это ваш реестр и ваша политика | Аккаунты мейнтейнеров, scoped-пакеты, 2FA для публикации; экосистема инструментов provenance/signing развивается | Аккаунты на Packagist, привязка к VCS; `composer.lock` фиксирует конкретные коммиты/релизы | Аккаунты мейнтейнеров, wheel/sdist; lock-файлы и хеши в инструментах (`pip-tools`, Poetry, `uv` и др.) |
| **Self-host / приватный реестр** | Референс-сервер на Go, файловое хранилище, переменные `REGISTRY_*` (см. [README](../README.md)) | Verdaccio, npm Enterprise, Artifactory и др. | Satis, Private Packagist, зеркала | devpi, Nexus/Artifactory, внутренние индексы |
| **CLI** | `skillget` (Go), lockfile и разрешение версий в одном стеке с реестром | `npm`, `yarn`, `pnpm` | `composer` | `pip`, `pipenv`, `poetry`, `uv` … |
| **Зрелость экосистемы** | Ранняя, узкая предметная область (skills) | Очень высокая, десятилетия массового использования | Высокая для PHP | Очень высокая для Python |

## Сценарии использования (без перегиба)

| Сценарий | Где уместен skpkg | Где уместны классические менеджеры |
|----------|-------------------|-------------------------------------|
| Поставить зависимости **runtime** бэкенда на Node | Нет — это npm/yarn/pnpm | npm и аналоги |
| Поставить библиотеку для **PHP** или **Python** | Нет | Composer / pip и аналоги |
| Версионировать и доставлять **наборы правил и документов для ИИ-агентов** в одной команде, с lockfile под CI | Да — это целевой сценарий skpkg | Теоретически можно «хакнуть» через git submodules или tarball, но без специализированного контракта skill-пакета |
| Корпоративный **внутренний каталог** скиллов с аудитом публикаций | Да — self-hosted реестр + токен записи | Не primary; для кода обычно Artifactory/GitHub Packages и т.д. |

## Модель доверия (чуть подробнее)

- **npm / PyPI / Packagist** опираются на глобальные аккаунты мейнтейнеров, политики реестра, сканирование, репутацию пакетов и практики команд (lock-файлы, pinning, зеркала). Модель угроз изучена широко; инциденты с typosquatting и компрометацией аккаунтов — известный класс рисков.
- **skpkg** в текущем виде: доверие = **кто выдал токен записи** + **целостность артефакта (хеш)** + возможность **снять версию с установки (yank)**. Нет автоматического «социального графа» доверия как у крупных публичных реестров — зато политика полностью задаётся владельцем реестра (публичный каталог vs закрытый).

Для постов: корректная формулировка — «skpkg переносит привычные идеи **версионируемых артефактов и lockfile** в домен **skills**, а не заменяет npm/PyPI для приложений».

## Self-host

- **skpkg:** поднять `registry`, указать `REGISTRY_DATA_DIR`, при необходимости `REGISTRY_WRITE_TOKEN` для записи; клиенты ходят на ваш базовый URL API. Подробности в корневом [README](../README.md) и [registry-api.md](./registry-api.md).
- **npm / Composer / PyPI-стек:** зрелые гайды по приватным прокси и air-gapped средам; skpkg здесь **проще по поверхности**, но **меньше** готовых интеграций с корпоративным IAM «из коробки».

---

## CLI

- **skillget** — установка и разрешение версий skill-пакетов против выбранного реестра (см. [PUBLIC_ECOSYSTEM.md](./PUBLIC_ECOSYSTEM.md)).
- Сравнивать «удобство CLI» с `npm install` в массовом смысле **некорректно**: у skpkg другой объект (skills, не модули приложения) и другая стадия зрелости UX.
